# Tech Knowledge Card — 攻坚手札·大事记

## 核心理念

**不是讲"这个技术是什么"，而是还原"这个难题是怎么被啃下来的"。**

一项技术问题摆在面前，你不知道答案。你试了方案 A，不行。试了方案 B，还是不行。试了方案 C，又卡住了。直到某一步，你突然看清了问题本质——答案不是天上掉下来的，是**三败之中逼出来的**。

这篇手札不罗列知识点，而是以「大事记」的编年体格式，逐阵记录攻坚的每一步：你站在这座城前，看到了什么？你用了什么兵器？为什么攻城失败了？每次失败你学到了什么？最后城门怎么破的？破城之后又看到了哪些新的狼烟？

**读者读完的感受**：不是"哦，原来解法是这样"，而是"原来这么想就能找到解法，以后我也能这样想"。

## 触发条件

- 用户提到"攻克""攻坚""解决问题""怎么解决的"
- "层层探索""层层递进""试了不行""尝试失败"
- "踩坑""踩坑记录""血泪史""教训"

- 类似意图：想知道面对一个技术难题时的**完整思考过程**，不是只看结果，要看中间的失败和转折

## 视觉风格详述

与武侠漫画风共享同一套配色 DNA，但排版和组件完全不同：

- **封面**：「攻坚手札」标签 + 技术名词标题 + 「录·{{AUTHOR_NAME}}」录者署名
- **大事记时间轴**：左侧竖线 + 菱形节点（失败标朱砂红，突破标金色圆）
- **阵号标记**：每阵用「第一阵」「第二阵」「决胜局」标记，右侧配「败」「成」标签
- **核心组件**：
  - `.crisis-scene` 军情急报：问题场景还原
  - `.stake-box` 赌注框：不解决会怎样
  - `.chronicle-log` 大事记时间轴（竖线 + 菱形节点）
  - `.battle-entry` 大事条目：每阵包含 `尝试思路 → 沙盘推演 → 失败原因 → 学到什么` 四层递进
  - `.breakthrough-box` 破阵时刻：金光高亮突破点
  - `.new-threat` 新狼烟：破城后的新问题
  - `.forward-path` 展望：下一城

## 内容构建：大事记叙述线

本技能不讲"是什么、为什么用、怎么用、有什么问题"的四维分析。**本技能的核心是：一个工程师面对一个具体技术问题，用层层递进的尝试把它啃下来。**

下面按四回目的结构组织内容（供前端程序员实现模板时一一对应）：

---

### 壹回 · 兵临城下 → 【问题】这座城长什么样？

> **目标**：让读者站在你的位置，看到同样的困难，感到同样的压力。

**写作要求**：

1. **场景还原**：用真实可感的场景开场——不是"系统面临高并发挑战"，而是"凌晨三点，报警响了，CPU 飙到 95%，日志刷屏。我盯着 Grafana，脑子一片空白。"
   
2. **问题定义**：用一句话说清核心矛盾——「A 和 B 不能同时满足，但业务要求两者必须兼得」这种格式。

3. **已有方案为什么不够用**：市面上有什么解法？为什么在这个具体场景下它们全都不行？——不是它们不好，是它们的假设条件在这个场景下不成立。

4. **赌注**：不解决会怎样？——不要夸张，说真实损失。宕机赔钱就写赔钱，用户体验崩就写崩。真实感比戏剧性更重要。

5. **结尾留钩子**：「我决定动手了。那时我还不知道，前面有三座城要一座一座地破。」

**「军情急报」框**：放核心矛盾的一句话描述。  
**「若不破此关」框**：放不解决的后果。

---

### 贰回 · 逐鹿破阵 → 【探索】大事记 —— 一计不成再生一计

> **目标**：这是本技能最核心的一回。读者跟着你一起试、一起败、一起学。败了三次，但每次败都离真相近一步。

**写作要求**：至少 3 阵递进。每阵用统一结构：

| 组件 | 写法 |
|------|------|
| **阵号 + 标签** | `第一阵 / 第二阵 / 第三阵` + `败`/`成` 标签 |
| **思路** `.be-attempt` | 这一阵的核心思路——为什么你觉得这能行？说直觉、说推理、说假设。 |
| **沙盘推演** `.sandbox-box` | 这个方案的技术逻辑。不用完整代码，用推理链：因为 X，所以 Y，应该能 Z。 |
| **为什么败了** `.be-failure` | 具体、可复现的失败原因。不要"性能不够"这种空话，要说"在 10 万 QPS 下，锁竞争导致 80% 时间花在等待上"。最好有一个最小可感的坏场景。 |
| **悟到的** `.be-lesson` | 这次失败教给你什么？必须和下一阵的思路**有因果链条**——下一阵为什么那样试？因为这一阵让你看清了某件事。 |

**递进要求**：
- 第一阵和第二阵之间的递进：第一阵暴露出某个假设是错的 → 第二阵换了一个假设
- 第二阵和第三阵之间的递进：第二阵发现不是假设问题，是架构层面的问题 → 第三阵重新设计架构
- 每一阵都比上一阵**更接近问题本质**，读者能感受到"虽然败了，但认知在升级"
- 不要写"试了 A 不行，换了完全不相关的 B 也不行"——这不是探索，这是乱撞

**失败原因的专业度**：
- 用真实的技术术语说真实的失败原因
- 如果涉及性能，给出具体数字（即使是大致数量级）
- 如果涉及架构，画出问题出在哪个节点/哪个层
- "专业度不减"——不能因为是武侠风就泛泛而谈

**字数建议**：每阵 200-400 字。

---

### 叁回 · 一剑封喉 → 【突破】第四次终于成了

> **目标**：让读者感受到「啊，原来如此！前面三次失败原来都是在铺路！」

**写作要求**：

1. **决胜局思路** `.battle-entry.breakthrough`：最终方案的核心思路——**必须说清它和前三阵的关系**。不是说"灵机一动突然想到"，而是"前三阵加起来等于告诉你：问题不在 X 而在 Y，所以正确解法应该是 Z"。

2. **破阵时刻** `.breakthrough-box`：描述突破瞬间——你看到了什么证据让你确信这次对了？是压测数字？是单元测试全绿？还是架构图上某个矛盾终于消解了？

3. **利器清单** `.arsenal-list`：提炼 2-3 个核心设计决策。每个决策要回答：
   - 它解决了前面哪一/哪几阵暴露的问题？
   - 如果不用这个设计，会有什么后果？
   - 这个设计的取舍是什么？（好的设计都有代价，说清楚）

4. **总结**：用一句话说出这道题的关键——让读者以后遇到类似的题能想起这句话。

---

### 肆回 · 余波未平 → 【新问题】城破了，新狼烟起了

> **目标**：不让读者产生"问题完美解决了"的错觉。真实工程中，每个解决方案都会引入新问题——诚实面对，才是真正的专业。

**写作要求**：

1. **新狼烟** `.new-threat`：解决方案引入的新问题或局限性，2-3 个：
   - 不要求"必然出现"，可以是"在某些边界条件下会暴露"
   - 不过度渲染负面，但也不粉饰太平
   - 每个问题要具体：什么场景下出现？临界条件是什么？

2. **展望** `.forward-path`：这套方法论的价值——不只是解决了这一个问题，更是一套可复用的思考框架。下一次遇到新题，怎么用这套"三败出真知"的路子。

3. **攻坚者旁白**：结尾点睛——"技术攻坚的本质不是找到完美答案，是在一堆不完美答案里，找到那个**问题问对了**的方向。"

---

### 攻坚者旁白 ⚔

- **位置**：各回转折处，每回至少 1 条
- **内容**：模仿当事工程师的语气，来一句事后总结或吐槽：
  - 失败后："当时觉得天塌了，现在想想——幸亏那一阵败了，不然我永远不会去想那个方向。"
  - 突破后："不是突然开窍，是前三阵已经把错误答案全排除了。"
  - 新问题前："城破了就以为完事了？想得美。后面还有一百座城。"
- **语气**：干练、有点幽默但不轻浮、像一个老兵的战后复盘

## 截图与切图流程

### 移动端 HTML → PNG 长图

使用 Playwright 渲染移动端 HTML 为完整长图：

```bash
python -c "
from playwright.sync_api import sync_playwright
with sync_playwright() as p:
    b = p.chromium.launch()
    page = b.new_page(viewport={'width': 420, 'height': 800})
    page.goto('file:///path/to/output/{name}_wuxia_conquest.html')
    page.screenshot(path='{name}_wuxia_conquest.png', full_page=True)
    b.close()
"
```

### 长图切割

使用随技能包附带的 `scripts/html_slice.py`：

```bash
python scripts/html_slice.py output/{name}/{name}_wuxia_conquest.html --card-height 1500
```

切完后验证每张图宽度一致。

## 运营策略适配（涨粉导向）

### 标题情绪设计
- **好奇心 + 获得感** 为第一驱动力：标题要暗示"里面有真实的失败和转折"
  - 示例："ZooKeeper 脑裂问题，我花了三天、败了三阵、才想通"
  - 示例："这个问题困了我两年，解法简单到你不敢相信"
- **社交货币**叠加："下次面试被问怎么解决问题的，直接把这篇甩给他"
- **共鸣**开场："每个工程师都有一段'怎么试都不行'的至暗时刻"

### 互动设计
- 在每次"败了"的环节设悬念："如果是你，你会怎么改？评论区写下你的答案"
- 文末引导："你工作中遇到过什么'一计不成再生一计'的时刻？"
- "把这篇文章转给正在焦头烂额的同事——告诉他，你不是一个人在死磕"
